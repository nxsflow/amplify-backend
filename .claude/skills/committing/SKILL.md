# Committing in amplify-backend

## Prerequisites

Before committing, ensure you're on a feature branch, not `main` or `upstream`.

```bash
git branch --show-current
```

If on `main` or `upstream`, create a branch first. See the `branching-strategy` skill for which base to use.

Branch naming: `<type>/<short-description>` (e.g., `feat/add-user-auth`, `fix/null-pointer-error`)

## Commit Message Format

```
<type>(<scope>): <subject>

<body>

<footer>
```

The header is required. Scope is optional. All lines must stay under 100 characters.

### Commit Types

| Type | Purpose |
|------|---------|
| `feat` | New feature |
| `fix` | Bug fix |
| `chore` | Maintenance, deps |
| `refactor` | Refactoring (no behavior change) |
| `perf` | Performance improvement |
| `docs` | Documentation only |
| `test` | Test additions or corrections |
| `ci` | CI configuration |
| `revert` | Revert a previous commit |

### Subject Line Rules

- Imperative, present tense: "Add feature" not "Added feature"
- Capitalize the first letter
- No period at the end
- Maximum 70 characters

### Body Guidelines

- Explain **what** and **why**, not how
- Imperative mood, present tense
- Include motivation for the change
- Contrast with previous behavior when relevant

### Footer

Reference issues:
- `Fixes #1234` â€” closes the issue on merge
- `Refs #1234` â€” links without closing

### Examples

```
fix(auth): Handle null response in user endpoint

The user API could return null for deleted accounts, causing a crash
in the dashboard. Add null check before accessing user properties.

Fixes #5678
Co-Authored-By: Claude <noreply@anthropic.com>
```

```
feat(data): Add Slack thread replies for alert updates

When an alert is updated or resolved, post a reply to the original
Slack thread instead of creating a new message. This keeps related
notifications grouped together.

Refs #1234
```

```
feat(api)!: Remove deprecated v1 endpoints

Remove all v1 API endpoints that were deprecated in version 23.1.
Clients should migrate to v2 endpoints.

BREAKING CHANGE: v1 endpoints no longer available
Fixes #9999
```

```
revert: feat(api): Add new endpoint

This reverts commit abc123def456.

Reason: Caused performance regression in production.
```

## AI-Generated Changes

When changes were primarily generated by a coding agent, include in the footer:

```
Co-Authored-By: Claude <noreply@anthropic.com>
```

This is the only AI indicator that should appear in commits. Do not add "Generated by AI" or similar markers elsewhere.

## Pre-commit Hook

When you commit, Husky runs automatically:

1. `npm run update:create-amplify-deps` â€” updates create-amplify dependency metadata
2. `npm run update:tsconfig-refs` â€” updates TypeScript project references
3. `npm run check:package-lock` â€” validates package-lock.json integrity
4. `npx lint-staged` â€” on staged files:
   - `*.ts`: prettier --write + eslint --cache --fix
   - `*.json`, `*.yml`, `*.md`: prettier --write

**Expect**: The hook may modify files (formatting, ref updates). These modifications are auto-staged.

**If the hook fails**: Fix the reported issue and try again. Do NOT use `--no-verify`.

## Pre-push Hook

When you push, Husky validates:

1. `changeset status --since main` â€” ensures changesets exist for changes
2. `npm run check:api` â€” validates public API surface

**If changeset check fails**: Run `npx changeset` to create one. See the `changesets` skill.
**If API check fails**: Run `npm run update:api` to regenerate API reports, then commit them.

## Staging Recommendations

- Stage specific files, not `git add -A`
- Never commit `.env`, credentials, or secrets
- The pre-commit hook will auto-format staged files â€” review the diff after committing

## Principles

- Each commit should be a single, stable change
- Commits should be independently reviewable
- The repository should be in a working state after each commit
